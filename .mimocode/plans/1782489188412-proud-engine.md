# pytest-llm: Comprehensive Project Summary

## Executive Overview

**pytest-llm** is an open-source pytest plugin that provides LLM-powered semantic assertions for testing AI-generated text outputs. Instead of brittle string matching, it uses LLM judges and local embeddings to evaluate faithfulness, tone, safety, hallucinations, and more.

- **Package name**: `pytest-llm-sushit` (PyPI)
- **Version**: 0.1.0 (Alpha status)
- **License**: MIT (2025)
- **Author**: Sushit
- **Python support**: 3.9+
- **GitHub**: https://github.com/Sushit-prog/pytest-llm

## Project Structure

The repository contains **two distinct components**:

### 1. Core pytest-llm Plugin (`src/pytest_llm/`)
The main product - 6 files forming the pytest plugin:

| File | Purpose |
|------|---------|
| `__init__.py` | Public API exports (8 assertions + JudgeResult + LLMJudge + LLMReporter) |
| `judge.py` | LLMJudge class with multi-provider support, 3-retry logic, caching, embeddings |
| `assertions.py` | 8 assertion functions with dedicated system prompts |
| `plugin.py` | pytest plugin: CLI options, `llm_judge` fixture, LLMReporter integration |
| `config.py` | `pytest_configure_judge()` for conftest.py configuration |
| `reporter.py` | Rich-based summary table of assertion results |

### 2. AI Reliability Platform (`app/`)
A separate FastAPI web application (merged from diverged branch) - **out of scope for pytest-llm**:
- Dashboard with eval runs, trace viewer, provider analytics
- SQLite database via SQLModel
- Async provider implementations (Groq, Mistral, Gemini, OpenRouter)
- Docker deployment support

## Core Features Built

### 8 Assertion Functions

| Function | Mechanism | Uses API? | Description |
|----------|-----------|-----------|-------------|
| `assert_faithful` | LLM judge | Yes | Every factual claim supported by source |
| `assert_no_hallucination` | LLM judge | Yes | No invented facts |
| `assert_tone` | LLM judge | Yes | Matches expected tone |
| `assert_semantic_similarity` | sentence-transformers | No | Cosine similarity vs expected text |
| `assert_contains_claim` | LLM judge | Yes | Semantically contains a claim |
| `assert_safe` | LLM judge | Yes | No harmful content |
| `assert_language` | LLM judge | Yes | Written in expected language |
| `assert_regression` | Dual (cosine + LLM) | Yes | Not worse than a baseline |

### LLMJudge Class (`judge.py`, 221 lines)
- **Multi-provider support**: OpenAI, Anthropic, Groq, Ollama
- **Provider selection**: Environment variables (`LLM_JUDGE_PROVIDER`, `LLM_JUDGE_MODEL`) or CLI flags
- **Lazy client initialization**: Only imports SDK when needed
- **3-retry logic**: With graceful degradation to `passed=False, score=0.0`
- **Response parsing**: Handles markdown code fences, JSON extraction, score clamping [0.0, 1.0]
- **JudgeCache**: SHA-256 hashed caching (key: `provider:model:system_prompt:user_prompt`)
- **Local embeddings**: sentence-transformers (`all-MiniLM-L6-v2`) for cosine similarity

### Configuration Hierarchy
1. **CLI flags**: `--llm-judge-provider`, `--llm-judge-model`, `--llm-report`
2. **conftest.py**: `pytest_configure_judge(provider=..., model=...)`
3. **Environment variables**: `LLM_JUDGE_PROVIDER`, `LLM_JUDGE_MODEL`, provider-specific API keys

### pytest Plugin Integration
- Auto-loaded via `pytest11` entry point in `pyproject.toml`
- Session-scoped `llm_judge` fixture for shared client instance
- `LLMReporter` for Rich-based pass/fail/score/reason tables
- CLI options for runtime configuration

## Known Issues & Gaps

### Architecture
1. **Dual architecture confusion**: Two distinct projects in one repo (pytest-llm plugin + AI Reliability Platform)
2. **Provider duplication**: Groq client exists in both sync (test side) and async (app side) implementations
3. **Tracing duplication**: Three different tracing pathways create architectural confusion

### Implementation Gaps
1. **System/user prompt fusion**: Prompts fused into single `user`-role message instead of separate roles
2. **No retry backoff**: Retry loop has no delay between attempts
3. **Unbounded cache**: No TTL or size limit on JudgeCache
4. **Hardcoded prompts**: No custom system prompt override for assertions
5. **Binary assertions**: 4 assertions use binary `result.passed` without tunable sensitivity

### Testing
1. **No integration tests**: Only mock judges, no real provider testing
2. **No app/ tests**: Web dashboard has no test suite

### Deployment
1. **No PyPI publish workflow**: No GitHub Actions step for publishing
2. **Incomplete error handling**: Uses `print()` + `traceback.print_exc()` instead of structured logging
